# Strata - Architecture

## Overview

Strata is a creator-economy transcription marketplace built on [Distributive's DCP (Distributive Compute Protocol)](https://distributed.computer) network. It is the hackathon entry for **BearHacks 2026** -- a 36-hour sprint (April 24-26, 2026) by a team of four (FE + BE1 + BE2 + BE3).

**Key innovation:** Whisper-WebGPU inference runs inside visitor browsers via DCP's distributed compute infrastructure. There is no central server performing transcription. Visitors to Distributor sites silently contribute idle compute cycles, and content creators get transcription at a fraction of the cost of centralized services.

**Three actors:**

- **Client** -- content creators (podcasters, YouTubers, course creators) who submit audio for transcription and pay per audio-hour.
- **Distributor** -- site owners who embed a Strata `<script>` tag and earn 68% of the revenue generated by their visitors' compute contributions.
- **Node** -- an anonymous visitor browser running the embed runtime. No signup, no account, no PII. The only visible surface is a footer chip with a "What is this?" link.

**Economic model:** Client pays USD -> Strata escrows DCC on DCP -> Distributor gets 68%, Strata keeps 32%. Nodes are unpaid; an ad-free browsing experience is the value exchange.

## Core Components

### 1. Strata Platform (Next.js 14.2.35, App Router)

The main web application serving both Distributor and Client dashboards from a single Next.js project. Role-gated layouts enforce access control via Auth0 custom claims (or stub-mode cookies in development).

- **Location:** `/strata/`
- **Key files:**
  - `src/app/` -- App Router with route groups for marketing, distributor, and client
  - `src/lib/auth.ts` -- Session management (`getSession()`, `requireRole()`)
  - `src/lib/db.ts` -- Prisma client singleton
  - `src/lib/sse.ts` -- In-memory SSE channel infrastructure
  - `src/lib/useJobStream.ts` -- Client-side React hooks for SSE consumption
  - `prisma/schema.prisma` -- Full data model
  - `proxy.ts` -- Auth0 v4 middleware mount

### 2. Embed System (Cloudflare Pages)

A two-part system deployed to `embed.strata.dev` that turns visitor browsers into DCP compute Nodes.

- **Location:** `/embed/`
- **`strata.js`** -- An IIFE loader (~2KB) that Distributors paste into their site via a `<script>` tag. It injects a hidden runtime iframe and renders a footer chip with pause/resume controls and status indicators.
- **`runtime.html`** -- The iframe content that loads `dcp-client` from the DCP scheduler, fetches per-slot configuration from the Strata API, creates a DCP Worker with Private Compute Group credentials, and starts pulling Slices.
- **`what-is-this.html`** -- An explainer modal page linked from the footer chip.

Communication between the IIFE chip and the runtime iframe uses `postMessage` with origin validation.

### 3. DCP Library (Node.js Whisper Transcription)

A standalone Node.js library that handles audio decoding, chunking, and distributed transcription over the DCP network.

- **Location:** `/dcp/`
- **Public API (4 functions):**
  - `decodeAudio(input)` -- Converts audio files to Float32Array at 16kHz mono via ffmpeg
  - `chunkAudio(samples, sec)` -- Splits samples into 30-second chunks
  - `transcribeChunks(chunks, opts)` -- Distributes chunks to DCP via `compute.for()`, runs Whisper-tiny ONNX inference on worker Nodes
  - `transcribe(input)` -- Convenience wrapper: decode + chunk + transcribe + join
- **Work function:** Self-contained (serialized via `Function.prototype.toString()`), loads ONNX Runtime + Whisper-tiny encoder/decoder models inside the DCP V8 sandbox, probes for WebGPU with WASM-SIMD fallback, and calls `progress()` to avoid ENOPROGRESS termination.

### 4. Music App (Proof-of-Concept Demo)

A Next.js application demonstrating the "compute-instead-of-ads" concept -- a music streaming app where listeners contribute idle compute rather than viewing advertisements.

- **Location:** `/music-app/`
- **Stack:** Next.js 14.2.5, React 18, TypeScript
- **Key files:** `lib/compute.ts` (DCP integration), `lib/settings.ts` (user preferences), `app/settings/page.tsx` (compute toggle UI)

## Data Models

The Prisma schema (`strata/prisma/schema.prisma`) defines the following models. The current implementation uses SQLite for development; production targets Neon Postgres.

### User
Core identity, linked 1:1 with either a Distributor or Client profile. Fields: `id`, `auth0Sub`, `email`, `role` (distributor | client | admin).

### Distributor
Site owner who earns revenue from visitor compute. Fields: `id`, `userId` (unique), `displayName`, `stripeConnectAccountId`, `dcpPaymentAddress`, `status`. Has many Sites, ComputeSlots, and Settlements.

### Client
Content creator who submits transcription jobs. Fields: `id`, `userId` (unique), `displayName`, `stripeCustomerId`, `balanceCents`. Has many Jobs.

### Site
A domain registered by a Distributor. Verified via `/.well-known/strata.json` token exchange. Fields: `id`, `distributorId`, `domain`, `verificationToken` (unique), `verified`. Has many ComputeSlots.

### ComputeSlot
A named embed placement on a verified Site. The `id` becomes the `data-slot` attribute in the embed snippet. Fields: `id`, `siteId`, `distributorId`, `name`, `active`.

### Job
A transcription job submitted by a Client (user-facing name: "Forecast"). Fields: `id`, `clientId`, `name`, `description`, `status` (queued | rollouts | verifying | done | failed), `workFnTemplate`, `inputSetConfig`, `inputCount`, `nRollouts`, `useVerifier`, `budgetCents`, `perSliceCents`, `dcpJobId`, `dcpVerifierJobId`, `pcgJoinKey`, `pcgJoinSecret`. Has many Slices and Settlements.

### Slice
One unit of work within a Job -- typically a 30-second audio chunk. Fields: `id`, `jobId`, `index`, `phase` (rollout | verifier), `status` (pending | claimed | completed | failed), `nodeSession`, `resultHash`, `resultData`, `completedAt`.

### Settlement
Revenue distribution record created when a Job completes. Implements the 68/32 split. Fields: `id`, `jobId`, `distributorId`, `slotId`, `grossCents`, `distributorCents` (68%), `strataCents` (32%).

### Key Relationships

```
User 1:1 Distributor 1:N Site 1:N ComputeSlot
User 1:1 Client 1:N Job 1:N Slice
Job N:1 Client
Settlement N:1 Job, N:1 Distributor
```

### Planned Models (from Architecture Doc, Not Yet in Schema)

The architecture document specifies additional models that are part of the target design but have not yet been implemented in the Prisma schema:

- **Forecast** -- Replaces/extends Job with fields for `inputManifestUrl`, `audioHoursTotal`, `languageScope`, `outputFormats`, `webhookUrl`, `workFunctionVersion`, `frontOpenedAt`, `sealedAt`
- **Catchment** -- Assembled output bundle per Forecast: `bundleUrl`, `audioHoursSealed`, `slicesCompleted`, `slicesTotal`, `sealedAt`
- **Attestation** -- Per-Slice signed receipt: `sliceId` (unique), `nodePubkey`, `nodeRegionGlyph`, `outputHash`, `schedulerSig`, `ts`

## Data Flow

The complete lifecycle from Distributor onboarding through to payout follows ten phases:

### 1. Distributor Onboarding

```
Distributor signs up (Auth0) -> Adds site domain -> Strata generates verification token
-> Distributor places /.well-known/strata.json on their domain
-> Strata fetches and verifies token match -> Site marked verified
-> Distributor creates ComputeSlot -> Receives embed <script> snippet
```

### 2. Embed on Distributor's Site

The Distributor pastes the `<script>` tag into their site template. The IIFE loader (`strata.js`) runs on page load, injects the hidden runtime iframe, and renders the footer chip.

### 3. Visitor Passive Compute

A random visitor lands on the Distributor's site. The runtime iframe fetches per-slot config from the Strata API, joins the Private Compute Group, starts the DCP Worker, and begins pulling available Slices from the DCP scheduler. The footer chip shows "Computing..." with progress.

### 4. Client Submits Forecast

A Client (content creator) signs into the Client dashboard, drops audio into the Forecast Composer (RSS feed URL, YouTube channel URL, or direct file upload). Optionally, Gemma 4 1B running in browser-WebGPU translates plain-English instructions into a Forecast spec. The audio is sliced into 30-second chunks.

### 5. Submit Worker Dispatches

The DCP submit worker (Node.js, running on Vultr or local with ngrok) picks up the Forecast from the database, builds an `inputSet` from the chunk manifest with k=2 redundancy (each chunk dispatched to two independent Nodes), and calls `compute.for()`. The job is assigned to the Private Compute Group.

### 6. Nodes Execute Whisper

Browser Nodes in the Compute Group pick up Slices from the scheduler. Each Node loads the Whisper-base ONNX model via WebGPU (or Whisper-tiny via WASM-SIMD fallback), decodes the 30-second audio chunk, runs inference, and returns the transcript text plus a semantic hash. The work function calls `progress()` throughout to maintain the heartbeat.

### 7. Quorum + Oracle

Results arrive at the submit worker via `job.on('result')`. For each chunk, the server compares semantic hashes from the two independent attempts (k=2). Matched pairs collapse to one accepted Slice. A 1-3% random sample is re-run against a server-side Whisper oracle for spot-check verification. Mismatches trigger reissue.

### 8. Catchment Assembly

Accepted Slices are assembled in timestamp order (not arrival order) to form the complete Catchment. Output formats include SRT, VTT, JSON, and plain text. An Attestation log (per-Slice signed receipts) is included by default.

### 9. Settlement (68/32 Split)

When a Forecast seals (all Slices completed), Strata calculates the revenue split: 68% to the Distributor, 32% to Strata. Settlement rows are written to the database. The Distributor dashboard updates with earnings.

### 10. Payout

Distributors accumulate DCC earnings. Payout is via Stripe Connect to their bank account. (For the hackathon demo, Stripe runs in test mode with no real money flow.)

## Real-Time System (SSE)

Strata uses Server-Sent Events for all live dashboard updates. The SSE infrastructure is process-local (in-memory `Map<string, Set<Controller>>`), which works for the hackathon because Vercel pins SSE connections to a single serverless instance region.

### Infrastructure (`src/lib/sse.ts`)

Four exported functions:

- **`subscribeSSE(channel, controller)`** -- Registers a `ReadableStreamDefaultController` on a named channel. Returns an `unsubscribe` teardown function.
- **`broadcastSSE(channel, payload)`** -- JSON-encodes `payload` and enqueues it as an SSE `data:` frame to all controllers subscribed to `channel`. Silently removes controllers that have closed.
- **`sseHeaders()`** -- Returns standard SSE response headers (`text/event-stream`, `no-cache`, `keep-alive`, `X-Accel-Buffering: no`).
- **`ssePing(controller)`** -- Sends a comment line (`: ping {timestamp}`) to keep the connection alive through proxies. Called on a 15-second interval.

### Channels

- **`job:{jobId}`** -- Streams Slice completion, status updates, and job lifecycle events to the Client Forecast Detail page.
- **`distributor:{distributorId}`** -- Streams `earnings_tick` events to the Distributor SliceTicker component.

### Client Hooks (`src/lib/useJobStream.ts`)

- **`useJobStream(jobId, initial?)`** -- React hook that opens an `EventSource` to `/api/jobs/{id}/stream`, dispatches events through a reducer to produce `JobState`, and handles reconnection with exponential backoff (1s to 15s cap, max 6 retries). Exposes `{ state, connection }` where `connection` is `'connecting' | 'open' | 'reconnecting' | 'closed'`.

- **`useDistributorTicks(distributorId, initial?)`** -- React hook that opens an `EventSource` to `/api/distributors/{id}/stream`, accumulates `earnings_tick` events into a rolling buffer of the 50 most recent ticks. Same reconnection logic.

### Event Types

```typescript
// Job stream events
| 'hello'          | Connection established, echoes jobId          |
| 'job_accepted'   | DCP job dispatched, includes total slice count |
| 'status'         | Progress update: total, distributed, computed  |
| 'slice_complete' | One slice finished, includes result data        |
| 'slice_error'    | One slice failed, includes error message        |
| 'job_done'       | All slices completed, includes winners map       |
| 'job_failed'     | Job terminated, includes error                   |

// Distributor stream events
| 'hello'          | Connection established, echoes distributorId   |
| 'earnings_tick'  | Revenue event: amountCents                      |
```

## Authentication

Strata supports two authentication modes, toggled by the `AUTH_MODE` environment variable.

### Stub Mode (`AUTH_MODE=stub` or unset)

Used during development. A cookie-based session (`strata-stub-session`) stores a JSON payload with `userId`, `email`, and `role`. The `POST /api/auth/stub` endpoint sets this cookie. No external auth provider is contacted.

### Auth0 Mode (`AUTH_MODE=auth0`)

Production mode using Auth0 Universal Login via `@auth0/nextjs-auth0` SDK v4.19. Key details:

- Auth0 v4 mounts routes at `/auth/callback` (not `/api/auth/callback` as in v3)
- The `proxy.ts` file at the project root handles the Auth0 middleware mount
- A post-login Action adds a custom `account_type` claim to the ID token, used for role-gated layout routing
- Both modes produce the same `Session` shape from `getSession()`, so all downstream code (route guards, SSE auth, API handlers) works identically regardless of mode

### Access Control

`requireRole(session, role)` is a TypeScript assertion function that throws a `401 Unauthorized` if no session exists or `403 Forbidden` if the session role does not match. Used in all protected API routes and SSE streams.

## Development Phases

### Phase 0: Preflight (T-1 to T+0)

DCP keystore funded with 5,000+ DCC. Audio fixture (30-60 min) committed to `/fixtures/`. Whisper-WebGPU sandbox spike confirmed or fallback path selected.

### Phase 1: Skeleton + DCP Proof (Hours 0-6)

Next.js scaffold with Prisma schema and SQLite. Stub-mode auth. DCP `compute.for()` returns one Whisper transcript end-to-end. WebGPU loads Whisper-base in browser. Design system tokens and AppShell layout.

**Exit criteria:** Vercel preview deploys on push. Stub auth works. DCP returns one Whisper transcript. WebGPU loads Whisper-base in browser.

### Phase 2: Vertical Slices (Hours 6-14)

Distributor onboarding flow (add site, verify, get embed). Client dashboard and Forecast submission API. DCP Forecast pipeline with k=2 redundancy. Embed runtime IIFE + iframe deployed to Cloudflare Pages.

**Exit criteria:** Distributor can add site and get snippet. Client can submit a Forecast. DCP dispatches Slices. Embed loads and worker starts. Dashboards render.

### Phase 3: Quorum + Catchment Assembly (Hours 14-22)

Server-side quorum logic (semantic-hash compare on k=2 attempts, third-attempt retry on mismatch, 1-3% oracle spot-check). Auto-tier promotion. Capability-ceiling enforcement. SSE live streaming. Catchment assembly in timestamp order. Settlement calculation (68/32 split). Full end-to-end wiring.

**Exit criteria:** Demo path works end-to-end. Record backup video at this checkpoint.

### Phase 4: Polish + Demo Prep (Hours 22-30)

Real Auth0 v4 swap-in. Pre-baked Catchment fallback (replays canonical Forecast via SSE if DCP scheduler unreachable). Multiple Node simulation (6 tabs). Dashboard SSE polish (reconnection, loading states). 5x end-to-end dry runs.

### Phase 5: Demo Video + Submit (Hours 30-36)

Record 90-second demo video. Pitch deck and judge talking points. Devpost long-form description. Final dry-runs. Submit at T-35:50 (before 10:00 EDT deadline).

## Risks and Mitigations

### Risk 1: Whisper-WebGPU Cannot Load Inside DCP Sandbox (HIGH / Critical)

The DCP V8 sandbox may not support loading `@huggingface/transformers` v3, accessing `navigator.gpu`, or `OfflineAudioContext`. Every Slice would fail.

**Mitigations (priority order):**
- **Path A:** Strata-hosted version-pinned bundle loaded via DCP's allowed-fetch surface
- **Path B:** Content-addressed bundle (SHA in path) registered as RemoteDataPattern
- **Path C:** Sandbox `fetch()` to localhost transcription endpoint. Real DCP scheduler, real model on demo laptop GPU -- only the execution location changes

### Risk 2: DCP Scheduler Unreachable on Conference WiFi (Medium / Critical)

**Mitigations:**
- Phone hotspot for presenter laptop
- Pre-baked Catchment replay: full Slice/Attestation/Catchment set replayed via SSE with realistic timing. UI labels the fallback explicitly. No hidden synchronous mode.
- Pre-recorded backup video on second laptop

### Risk 3: Whisper Cold Start During Demo (Medium / Visible stall)

First Slice on a cold Node stalls 6-12s on model download (~150MB). ENOPROGRESS may kill slices.

**Mitigations:**
- Pre-warm all 6 demo browser tabs 60+ seconds before judging
- Work function calls `progress()` at fetch, model warm, mid-inference, and before return
- Fallback to pre-warmed tab if one goes cold

### Risk 4: DCP Keystore Not Funded (Low / Critical)

`job.exec()` throws `ENOFUNDS`.

**Mitigation:** Pre-fund at T-1h with 5,000+ DCC. Re-check at T+34. Fall back to pre-baked Catchment if depleted.

### Risk 5: Gemma 4 Translator Fails (Low / Cosmetic)

Browser-WebGPU Gemma 4 1B returns malformed JSON during demo.

**Mitigation:** `forecast-translator.ts` short-circuits on canonical demo phrases, returning cached Forecast spec instantly. Live demo never depends on model round-trip.

### Risk 6: Auth0 Callback Misconfigured on Vercel (Medium / Non-blocking)

SDK v4 uses `/auth/callback` (not `/api/auth/callback`).

**Mitigation:** Lock to one Vercel production URL. If broken at demo time, flip `AUTH_MODE=stub` and redeploy (~30s).

### Risk 7: SSE Connection Drops / Vercel 5-min Timeout (Low / Cosmetic)

**Mitigation:** EventSource auto-reconnects. Forecast Detail hits catch-up snapshot on remount. Demo fixture pinned to seal in under 5 minutes.

### Risk 8: Cross-Origin Iframe / CSP Issues (N/A for Hackathon)

Demo site has permissive CSP. Production would require documented CSP requirements for Distributors.

### Risk 9: Forecast Runs Too Slow with 6 Nodes (Medium / Demo overrun)

60 min audio = 240 cycles at k=2 = ~7 min wall-clock on 6 Nodes.

**Mitigation:** Pin demo fixture to ~30 min audio (~3-4 min wall-clock). Pre-warm all tabs. Pre-baked Catchment as fallback.

### Risk 10: Compute Group Secret Inaccessible (Low / Workers idle)

**Mitigation:** Test at T+10. Fallback: omit `computeGroups` and accept any DCP worker.
